# 부광솔라 홈페이지 시스템 v10.0

## v10.0 변경사항

| 요청 | 변경 내용 |
|------|-----------|
| 하늘/땅이 뒤집혀 보이는 문제 | **진짜 원인**: 지형 메쉬의 좌표·노멀 계산 자체는 정상이었습니다(실제 Three.js로 직접 시뮬레이션해 검증). 진짜 원인은 `#site3d-canvas` 컨테이너가 `min-height: 460px`(CSS, 배경색 `#cfe8ff` 하늘색)인데 반해 렌더러는 항상 `480px` 고정 높이로 캔버스를 그려서, 두 값이 미세하게 어긋나는 화면 환경에서 컨테이너 하단에 CSS 배경색(하늘색)이 그대로 노출된 것이었습니다. 이게 "3D로 그려진 하늘+땅 화면" 아래에 "노출된 배경 하늘색"이 추가로 깔려, 마치 하늘이 땅 아래로 내려간 것처럼 보였습니다. 컨테이너를 `height: 480px`로 렌더러와 정확히 동일한 고정값으로 맞춰 더 이상 빈 공간이 생기지 않도록 했습니다 |

> 참고: 이 문제를 추적하는 과정에서 한 차례 "삼각형 와인딩이 뒤집혔다"는 잘못된 가설로 인덱스 보정 코드를 추가했었으나, 실제 Three.js 라이브러리로 직접 노멀 벡터를 계산해 검증한 결과 원본 좌표 계산이 처음부터 정상이었음을 확인하고 그 보정 코드는 제거했습니다(불필요한 변경이 남아있지 않도록 최종본에서 깨끗이 정리했습니다).

| 2~3. 서버 배포용 디렉토리 재구성 | `frontend/index.html`, `frontend/css/`, `frontend/js/`, `frontend/assets/`를 모두 프로젝트 루트로 이동했습니다. 백엔드(`backend/analytics_api.py`)는 `__file__` 기준으로 루트 경로를 자동 계산하고, `/css`, `/js`, `/assets`를 각각 정적 마운트하며 루트(`/`)에서 `index.html`을 직접 서빙하도록 재작성했습니다. 실제로 서버를 띄워 `/`, `/css/style.css`, `/js/*.js`, `/assets/*`, `/api/assets` 전부 HTTP 200 응답을 직접 확인했습니다 |

| 2. 건물 위치 어긋남 | **근본 원인**: 지형 정점 좌표는 그리드 인덱스 비례(gu/gv)로 계산하고 건물은 `latLonToLocalMeters()`로 별도 계산해, 수학적으로는 근사치였지만 부동소수점 오차가 모서리로 갈수록 누적됐습니다. 지형 정점도 `latLonToLocalMeters()`로 직접 계산하도록 통일해 건물과 100% 동일한 변환 함수를 거치게 했습니다 |
| 3. 3D 화면 다층 분리 오류 | 패널·지형·건물이 서로 다른 좌표 기준을 쓰던 문제를 표준 격자 시스템(`PANEL_CELL_X/Z`) 도입으로 해결했습니다. 모든 패널이 동일 격자의 셀 중심에만 생성되어 항상 정렬된 상태를 유지합니다 |
| 4. 레이어 토글의 실제 의도 반영 | 이전의 "지형/건물/패널 표시 on-off"를 완전히 폐기하고, 의도하신 대로 **"패널 설치 위치: 토지 위(영농형) / 건물 위(옥상형)"** 두 모드로 재설계했습니다. 토지 위는 기존 다리 달린 영농형 구조물을, 건물 위는 각 건물 옥상 면적에 맞춘 옥상형 패널을 자동 배치합니다 |
| 5. 클릭 배치 레이아웃 깨짐 / 비정형 패널 | **근본 원인**: 클릭한 지점을 그대로 패널 중심으로 사용해 자동배치 격자와 다른 임의 위치에 패널이 생성됐습니다. 클릭 배치도 표준 격자에 스냅하도록 변경해 자동배치 패널과 항상 동일한 행·열에 정렬되며, 이미 패널이 있는 칸은 중복 생성을 방지합니다. 설치 모드(토지 위/건물 위)에 따라 클릭 가능한 대상도 자동으로 구분됩니다(건물 위 모드에서는 옥상만, 토지 위 모드에서는 토지만 클릭 가능) |

| 주소→좌표 변환 | 코드에 `GOOGLE_MAPS_API_KEY` 변수가 있었지만 빈 문자열이라 실제로는 항상 OpenStreetMap Nominatim만 동작 | 헤더의 "Google Maps 미연동" 버튼으로 키를 직접 입력하면 **Google Maps JavaScript API의 PlaceAutocompleteElement(신규 Places 위젯)**가 두 주소 입력창(통합수익분석/입지분석)에 실제로 부착됨. 입력 중 Google의 실제 자동완성 드롭다운이 표시되고, 선택 시 Google이 반환한 정확한 위경도가 좌표로 사용됨 |
| 데이터 결합 흐름 | 좌표 확보 후 처리 로직이 지오코딩 함수 내부에 섞여 있어 Google 콜백과 공유 불가 | `processRevenueLocation()`, `processSiteLocation()` 두 공통 함수로 분리. 버튼 클릭(Nominatim 경로)과 Google Autocomplete 선택(`gmp-select` 이벤트) 양쪽에서 동일한 함수를 호출 — 좌표를 어디서 얻었든 이후 NASA/Open-Meteo/Overpass 조회 로직은 완전히 동일하게 동작 |
| 키 미입력 시 | - | Google 키를 입력하지 않아도 기존처럼 텍스트 입력+버튼(Nominatim) 방식으로 정상 동작. 화면이 깨지지 않음 |

키 입력은 헤더의 "Google Maps 미연동" 버튼을 눌러 모달에서 진행하며, 입력한 키는 브라우저 세션 메모리에만 보관되고 서버로 전송되지 않습니다.

> Google Geocoding API(서버용)는 결제 연동된 키가 필요해 대신 발급할 수 없었지만, Maps JavaScript API의 Places 위젯은 프론트엔드에서 같은 키로 동작하며 별도 백엔드 호출 없이 "주소검색 → 좌표"를 완성합니다.

3D 입지분석에서 "구글맵 등고선이 구현 안 되고 더미로 보인다"는 문제를 해결했습니다. 기존엔 좌표값을 시드로 한 사인파 합성 지형과, 좌표 시드로 무작위 배치한 가상 건물 5개를 보여주고 있었습니다. 이를 모두 제거하고 실제 공개 데이터로 교체했습니다.

| 항목 | 변경 전 (v4.0) | 변경 후 (v4.1) |
|------|------|------|
| 지형 | 좌표를 시드로 한 사인파 합성 더미 지형 | **Open-Meteo Elevation API**(Copernicus DEM, 90m 해상도) 실측 고도 그리드를 조회해 메쉬 생성 |
| 건물 | 좌표 시드로 무작위 배치한 가상 박스 5개 | **OpenStreetMap Overpass API**로 실제 반경 내 건물 폴리곤 + 실제 높이(`height`/`building:levels` 태그)를 조회해 압출(Extrude) 생성 |
| 데이터 소스 표기 | "Google 지형·건물"(실제로는 미사용) | "Open-Meteo(실측 지형) · OpenStreetMap(실제 건물) · NASA(일사량)"으로 정확히 표기 |
| 데이터 부재 시 동작 | 항상 더미를 그려 실데이터처럼 보임 | 주소 검색 전엔 평지만 표시, 실패 시 에러 메시지로 명확히 알림 (더미로 대체하지 않음) |

두 API 모두 키 발급이 필요 없는 무료 공개 API이며 브라우저에서 직접 호출 가능합니다(CORS 지원).

> Google Maps Elevation API/Building Insights(Solar API)는 결제 계정이 연결된 Google Cloud 프로젝트에서 키를 발급해야 하는 유료 API라 본 시스템에서 대신 발급할 수 없습니다. 실제 키를 보유하신 경우 `js/geocoding.js`의 `fetchElevationGrid()`, `fetchNearbyBuildings()` 두 함수만 Google API 호출로 교체하면 동일한 인터페이스로 더 정밀한 데이터를 쓸 수 있습니다.

## 화면 구성

## 통합수익분석 사용법 (주소 → NASA → 수익분석)

1. 주소 검색창에 설치 부지 주소 입력 후 "좌표 검색" 클릭 (OpenStreetMap Nominatim 오픈 API, 키 불필요)
2. 좌표가 확정되면 그 좌표로 NASA POWER 최근 1개년 월별 일사량을 조회
3. 평균 일사량(kWh/m²/day)을 등가 발전시간(h)으로 환산해 "일평균 발전시간" 입력값에 자동 대입
4. 그 외 PV/BESS/금융/단가 입력값은 "수익분석Profit Analysis.html"과 동일한 구조이며 자유롭게 수정 가능
5. "23년 수익 분석 실행"으로 결과 확인, "CSV 저장"으로 다운로드

주소 검색 없이도 기본값(발전시간 3.4h)으로 분석을 바로 실행할 수 있습니다.

## 입지분석 사용법 (주소+면적 → 실측 지형/실제 건물/NASA 결합 3D, 축적 1:20~1:5000)

1. 입지분석 화면에서 주소(전세계 가능)와 설치 부지 면적(평/m²)을 입력 후 "좌표 검색" 클릭
2. 좌표가 확정되면 그 위치에 빨간 핀 마커가 표시되어 정확한 검색 좌표를 확인할 수 있음
3. NASA 1년치 평균 일사량 조회 → Open-Meteo에서 그 좌표 주변 9x9 실측 고도 그리드 조회 → OpenStreetMap Overpass API에서 부지 면적에 비례한 반경 내 실제 건물 폴리곤과 높이를 순차 조회
4. 조회된 실측 데이터로 지형 메쉬와 건물 매스(있는 경우)가 먼저 생성되고, 그 다음 현재 선택된 설치 위치 모드에 맞춰 패널이 표준 격자에 자동 배치됨. 지형·건물·패널 모두 동일한 좌표 변환 함수를 사용해 위치가 정확히 일치함
5. 컨트롤 패널의 "패널 설치 위치"에서 **토지 위(영농형)**: 부지 중앙 격자에 다리 달린 패널 구조물을 자동 배치 / **건물 위(옥상형)**: 주변 건물 옥상 면적에 맞춰 패널을 자동 배치 — 두 모드 중 선택해서 전환 가능. 건물 위 모드에서 주변에 OSM 건물 데이터가 없으면 그 안내가 표시됨
6. "클릭으로 패널 배치하기" 버튼을 켠 뒤 3D 화면을 직접 클릭하면 표준 격자에 스냅되어 패널이 추가됨 — 토지 위 모드에서는 토지만, 건물 위 모드에서는 건물 옥상만 클릭 가능하며, 이미 패널이 있는 칸은 중복 생성되지 않음(초록/빨강 링으로 성공·실패 표시)
7. 컨트롤 패널 상단의 축적 버튼(1:20 / 1:50 / 1:100 / 1:1000 / 1:5000)으로 보기 배율을 전환 — 1:20~1:100은 패널·건물을 가까이서 보는 시공 디테일 뷰, 1:1000~1:5000은 지형·인접 건물 배치를 넓게 보는 광역 입지 뷰
8. "태양 시간대" 슬라이더로 그림자 이동을, "시점 회전" 슬라이더로 카메라 각도를 조절
9. 정보 패널에서 선택 위치, 분석 범위(축적), NASA 평균 일사량, 실측 지형 고저차, 그림자 영향률, 실효 발전효율을 확인

해당 지역에 OpenStreetMap 건물 데이터가 없으면(농지·산지 등) 건물 없이 지형만 표시되며, 이는 정상 동작입니다.

> 더 정밀한 데이터가 필요하면 Google Maps Elevation API, Building Insights(Solar API) 키를 발급받아 `geocoding.js`의 `fetchElevationGrid()`와 `fetchNearbyBuildings()` 두 함수만 해당 API 호출로 교체하면 됩니다. 반환 형식을 유지하면 `site3d.js`는 수정할 필요가 없습니다.

## 통화 선택 사용법 (원화/달러)

통합수익분석 화면 상단의 통화 선택 카드에서 "원화(KRW)" 또는 "달러(USD)" 버튼을 누르면, 23년 결과 테이블·차트·CSV 다운로드가 즉시 그 통화로 환산되어 표시됩니다. 계산 자체는 입력값(SMP/REC/CDM 단가 등)이 모두 원화 기준이므로 항상 원화로 수행되고, 통화 전환은 표시 시점에만 적용되어 계산 정확도에 영향을 주지 않습니다. 달러 선택 시 사용된 실시간 환율이 안내 문구에 함께 표시됩니다.

## 주소검색 전세계 동작 안내

통합수익분석·입지분석의 주소 검색은 1차로 Nominatim, 실패 시 2차로 Photon을 자동으로 시도합니다. 두 서비스 모두 키가 필요 없고 국가 제한이 없습니다. 매우 모호하거나 비표준 표기의 주소는 그래도 실패할 수 있으니, 이 경우 다음을 시도해보세요.

- 영문 표기로 입력 (예: "Tashkent, Uzbekistan")
- 도시/국가명까지 포함해 더 구체적으로 입력
- 건물명보다 도로명+번지를 우선 입력

## assets 폴더 자동 반영 (images / patents / videos)

`assets/images`, `assets/patents`, `assets/videos` 폴더(프로젝트 루트 바로 아래)에 파일을 새로 넣거나 빼기만 하면, 백엔드가 그 폴더를 직접 스캔해서 목록을 만들고 프론트엔드가 그 목록으로 갤러리·특허 인증서·기술 영상 화면을 그립니다. 코드를 수정하거나 다시 배포할 필요가 없습니다.

- 지원 확장자: 이미지/특허 `.jpg .jpeg .png .webp .gif`, 영상 `.mp4 .webm .mov`
- 표시 라벨은 파일명에서 확장자를 빼고 언더스코어·하이픈을 공백으로 바꿔 자동 생성됩니다 (예: `대전_시공사례_2026.jpg` → "대전 시공사례 2026")
- 최신 업로드(수정시각 기준)가 먼저 보이도록 정렬됩니다
- 백엔드(`backend/analytics_api.py`)가 실행되어 있어야 동작합니다. 백엔드가 꺼져 있으면(정적 호스팅 등) 기존 고정 파일명 추측 방식으로 자동 전환되어 화면이 비지 않습니다

API 엔드포인트: `GET /api/assets` (전체), `GET /api/assets/{images|patents|videos}` (개별)

## 접속 국가 기반 자동 언어 전환 (전세계 언어)

키 없는 IP 위치 조회(ipapi.co)로 접속 국가를 감지하고, Google Translate 엔진으로 그 국가의 공용어로 페이지 전체를 자동 번역합니다. 7개 언어로 고정되어 있던 이전 방식과 달리 Google Translate가 지원하는 90여 개 언어 전체가 대상이며, 위젯의 화면상 UI(드롭다운, 배너)는 보이지 않고 번역만 자동 적용됩니다.

한국에서 접속하면 번역 없이 원문(한국어) 그대로 표시됩니다. IP 조회가 실패하면(네트워크 차단 등) 브라우저 언어 설정으로 대체합니다.

지원 국가 매핑은 `js/i18n.js`의 `COUNTRY_LANG_MAP`에서 확인·확장할 수 있습니다.

## 홈 화면 YOLO 대화형 안내

`analytics.js`가 실시간으로 추정하는 방문자 성향(투자자/농업인/개발사/연구자/일반) 결과를 받아, 그 성향에 맞는 질문 카드 3~4개를 홈 화면에 보여줍니다. 카드를 클릭하면 질문과 관련된 화면(통합수익분석/입지분석/기술·특허/문의)으로 이동하며, 일부 카드는 입력값을 미리 채워줍니다. 방문자의 행동이 누적될수록(클릭, 체류시간 등) 성향 추정이 갱신되고, 홈 화면에 머무는 동안 8초마다 안내 카드가 자동으로 다시 그려집니다.

질문-답변 뱅크는 `js/main.js`의 `YOLO_GUIDE_BANK` 객체에서 추가·수정할 수 있습니다.

## 실행 방법 (서버 배포 — 루트 구조)

이 폴더(`solar_system_v9_0/`) 전체를 서버에 올린 뒤, 어디서든 다음 한 줄이면 바로 가동됩니다.

```bash
pip install fastapi uvicorn
python backend/analytics_api.py
```

실행 후 `http://서버주소:8000/` 으로 접속하면 `index.html`이 바로 뜨고, CSS·JS·assets·API가 모두 같은 서버에서 함께 서빙됩니다. `frontend/`로 따로 들어갈 필요가 없으며, 폴더를 통째로 옮기거나 압축해서 다른 서버에 풀어도 `backend/analytics_api.py`가 자기 위치 기준으로 루트를 자동 인식하므로 똑같이 동작합니다.

> 정적 호스팅(assets 자동 반영 없이 화면만 띄우고 싶을 때)만 필요하면 이 폴더에서 `python -m http.server 3000`을 실행해도 되지만, 이 경우 `/api/assets`가 없어 갤러리·특허·영상이 고정 파일명 추측 방식으로 자동 전환됩니다.
